Implementation Approach

What to Build First

Start with the dashboard, not a landing page. The first screen should already look and behave like Trackspace: a top status strip, a header, a left rail, a main workspace, and a drawer for details.

Do not paste the bundled mock into the app. Rebuild it as normal React components, with typed data and CSS that the project can maintain.

Do This

  • Use the mock for layout, spacing, colors, and behavior.
  • Keep src/app/page.tsx small.
  • Move the interactive app into a client component.
  • Use seed data first so the UI can move quickly.
  • Use Convex for live data once the screen design is stable.

How It Should Look

The mock is dark, compact, and data-heavy. Keep the thin borders, small labels, monospace numbers, status colors, and tight panel spacing. Avoid large marketing sections, oversized cards, and friendly startup-style copy.

Next.js Note

This project uses Next.js 16. Before changing app routes, layouts, metadata, route handlers, or server/client component boundaries, read the relevant guide in node_modules/next/dist/docs/.

Client Component Rule

The dashboard needs local state for navigation, filters, selections, and the drawer. Put that state in a "use client" component, then render it from the page.

Use Seed Data First

The mock already has the right kinds of records: capabilities, milestones, events, statuses, confidence labels, dependencies, and source links. Put those records into typed seed files first.

After the UI works, move the same shape into Convex so ingestion updates can stream into the app.

Capability Launch, landing, suits, power, comms, habitats, ISRU, and other required systems. Command Center, Dependency Map, Milestones

Step-by-Step Plan

Read the local Next.js docs

Check the relevant Next.js 16 docs before editing routes, layouts, metadata, server components, client components, or API routes.

Replace the starter page

This means changing the home route entry file, not building the whole UI yet. Remove the default create-next-app JSX from src/app/page.tsx: the Next logo, "To get started" text, Deploy button, and Documentation link. The page should only import the Trackspace app and render it. 

import { TrackspaceApp } from "@/features/trackspace/TrackspaceApp";
import "@/features/trackspace/styles/trackspace.css";

export default function Home() {
  return <TrackspaceApp />;
}

Add the app shell

This is the visible frame inside TrackspaceApp. Build the top strip, header, left rail, main screen area, and drawer. Step 2 wires the route to the app. Step 3 builds the frame the user actually sees.

Add typed seed data

Add statuses, confidence levels, capabilities, milestones, events, and helper functions for derived values.

Build the Command Center

Add the Earth/Moon scene, the readiness panel, blockers, recent changes, and next milestone rows.

Build the other screens

Add the dependency map, timeline, milestones, filters, selected states, and drawer links.

Move data into Convex

Once the UI is stable, define Convex tables, seed the same data, and subscribe from the app with live queries.

Compare with the mock

Open the app and docs/mock-design.html. Check layout, color, text size, spacing, scrolling, filters, and drawer behavior.

Checks Before Calling It Done

Check What Should Be True
Layout The first view has the top strip, header, left rail, main workspace, side panel, and drawer behavior shown in the mock.
Style Colors, borders, text sizes, labels, status chips, and panel spacing look close to the mock.
State Rail tabs, filters, selected nodes, selected events, selected milestones, and drawer open/close all work.
Data Readiness, status counts, blockers, related events, and graph edges come from the typed data, not copied display text.
Mobile The page is usable on a phone-size viewport. Text should not overlap, clip, or force the whole page sideways.
Avoid No starter UI, no marketing hero, no default graph styling, no placeholder boxes that do not match the mock.
